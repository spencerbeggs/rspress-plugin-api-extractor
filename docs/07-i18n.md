# i18n

The plugin works with RSPress's built-in internationalization. You declare your locales in the RSPress config and the generated API pages slot into the localized routing alongside your translated guides. The plugin configuration itself does not change for i18n — one `api` block covers every locale.

## Recipe

```ts
import path, { dirname } from "node:path";
import { fileURLToPath } from "node:url";
import { defineConfig } from "@rspress/core";
import { ApiExtractorPlugin } from "rspress-plugin-api-extractor";

const __dirname = dirname(fileURLToPath(import.meta.url));

export default defineConfig({
  root: "docs",
  title: "My Library",
  outDir: "dist",
  lang: "en",
  locales: [
    { lang: "en", label: "English" },
    { lang: "zh", label: "中文" },
  ],
  plugins: [
    ApiExtractorPlugin({
      logLevel: "info",
      api: {
        packageName: "my-library",
        model: path.join(__dirname, "api/my-library.api.json"),
        packageJson: path.join(__dirname, "api/package.json"),
        tsconfig: path.join(__dirname, "api/tsconfig.json"),
        theme: { light: "github-light-default", dark: "github-dark-default" },
      },
    }),
  ],
  route: { cleanUrls: true },
});
```

This mirrors the `i18n` example site. `lang` is the default locale, and `locales` lists every language with its switcher label. RSPress serves the default locale at the root and prefixes the others, so Chinese pages land at `/zh/...`.

## What the plugin localizes

The plugin generates the API pages once, and they appear under each locale's routing. The page UI strings that come from RSPress — the LLMs copy/open actions, the view-options labels — are localized through RSPress's own i18n. The API content itself (signatures, type names, summaries) comes verbatim from your `.api.json` model, so it reads in whatever language your source TSDoc comments are written in. To translate the documented API's prose, translate the TSDoc in your source and rebuild the model.

## Dead-link checking

When a locale does not yet have every page translated, RSPress's dead-link checker flags the gaps during the build. Relax it while translations are in progress:

```ts
export default defineConfig({
  // ...locales as above
  markdown: { link: { checkDeadLinks: false } },
});
```

The `i18n` example site sets this for exactly that reason.

## Combining with versioning

i18n composes with the [versioned recipe](./06-versioned.md). Declare both `locales` and `multiVersion`, and RSPress produces a route matrix of locale × version. The plugin config does not change — supply the per-version models through `versions` as usual.
